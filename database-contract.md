# Контракт базы слов «Bahasa»

Единый свод правил по словарю проекта. На него ссылается каждый чат, который читает или пополняет базу. Если правило меняется — меняем здесь, и только здесь.

## Коротко

Источник истины один — страница Notion **«📒 Bahasa Dictionary»**. Всё остальное производится из неё и руками не правится. Пополнение — только в Notion. Удаление — по обратной связи из приложения. У каждой записи есть постоянный `id`, который не меняется и не переиспользуется.

## Роли

- **Источник** — страница Notion «Bahasa Dictionary». Единственное место, которое правится вручную.
- **Сборщик** — чат-ассистент в проекте «Bahasa», который ведёт приложение: собирает `base.js` из Notion, назначает `id`, удаляет записи по запросу. Это роль, а не конкретный человек: её выполняет ассистент, работающий в проекте «Bahasa». Деплой готового `base.js` в GitHub сборщик **не** делает — на это нет доступа (см. роль пользователя).
- **Пользователь (владелец)** — заливает готовый `base.js` в репозиторий GitHub `bahasa-app` (у ассистента нет прав на запись в GitHub) и настраивает инструкции проектов.
- **Пополнители** — чаты-переводчики и тьютор (десктоп и мобилка): добавляют новые записи в Notion.
- **Приложение и пользователь** — инициируют удаление: приложение отдаёт список ненужных слов, пользователь передаёт его сборщику.

## Где что лежит

**Источник (единственный):** страница Notion «Bahasa Dictionary»
`https://app.notion.com/p/38cec2754c7e81188f86dcd80ada53a2`

**Производное + бэкап:** `lalabahasa-app/base.js`
Данные для приложения и одновременно полная резервная копия (содержит все поля). Генерируется сборщиком из Notion скриптом `build_base.py`. Никогда не редактируется вручную.

CSV удалён — он был лишней третьей копией и давал рассинхрон.

## Формат записи

Одна строка Notion = одна запись. Поля разделены ` · ` (пробел-точка-пробел):

```
- #id · **indonesian** · pronunciation · russian
```

Пример:
```
- #29 · **Ada kabel rol?** · а́да ка́бель-ро́л · Есть удлинитель?
```

Поля:

- **id** — постоянный числовой идентификатор.
- **indonesian** — слово или фраза (в `**жирном**`).
- **pronunciation** — русская транслитерация с ударением.
- **russian** — перевод. Варианты — через `;` или `/`.

Полей `type` (слово/фраза), `note` и `дата` в схеме нет — убраны как ненужные.

## Правила ID

- `id` назначает **сборщик** скриптом `build_base.py` — не пополнители и не вручную.
- Присваивается один раз и больше не меняется.
- **Никогда не переиспользуется**, даже после удаления: скрипт хранит наибольший когда-либо выданный номер в `base.js` → `LALA_NEXT_ID` (только растёт) и новым записям выдаёт следующий.
- Правка текста записи (опечатка, перевод) сохраняет `id`.

## Пополнение

Пополнители добавляют новые слова **только в Notion** — дописывают строку в раздел `## Записи`.

- `#id` у новой строки **не указывают** — его проставит сборщик при следующей сборке.
- Строка новой записи: `- **indonesian** · pronunciation · russian`.

## Удаление

Инициатор — приложение:

1. В приложении пользователь помечает ненужные слова кнопкой «🗑 Удалить из словаря»; список копируется в чате «Прогресс».
2. Пользователь передаёт список **сборщику** (в чат проекта «Bahasa»).
3. Сборщик удаляет соответствующие строки из Notion и пересобирает `base.js`.

Поскольку `id` постоянные, удаление одних записей не сдвигает номера других — прогресс в приложении не ломается.

## Сборка (Notion → приложение)

Шаги 1–3 выполняет **сборщик**, шаг 4 — **пользователь** (деплой в GitHub делает владелец репозитория; у ассистента нет прав на запись в GitHub).

1. Сборщик читает страницу Notion, копирует строки из `## Записи` в файл `notion_dump.txt`.
2. Сборщик запускает `python3 build_base.py notion_dump.txt` → обновляется `lalabahasa-app/base.js`, создаётся `notion_normalized.txt`.
3. Если при сборке появились новые `#id` (кто-то добавил слова без номера) — сборщик заливает `notion_normalized.txt` обратно в страницу Notion, чтобы номера зафиксировались. Если новых слов без номера не было — шаг пропускается.
4. **Пользователь** заливает новый `lalabahasa-app/base.js` в репозиторий GitHub `bahasa-app`. Приложение подхватит при следующем открытии.

Зачем шаг 3: пополнители добавляют слова без `id`; сборка присваивает номер только в `base.js`. Если не вернуть его в Notion, при следующей сборке запись снова окажется без номера и может получить другой `id` — а меняющийся `id` ломает прогресс в приложении. Возврат фиксирует номер в источнике навсегда.

## Инварианты (чего нельзя)

- Не редактировать `base.js` вручную — только генерировать.
- Не хранить базу нигде, кроме Notion (никаких новых CSV/копий как источников).
- Не менять и не переиспользовать `id` существующих записей.
- Пополнять и исправлять записи — только в Notion.

## Как подключить новый чат к этим правилам

Чтобы чат (тьютор, переводчик, приложение) знал контракт без ручного предупреждения — один раз добавить в инструкции соответствующего проекта строку-указатель:

> База слов «Bahasa» ведётся по контракту `database-contract.md` (папка проекта «Bahasa»). Источник истины — страница Notion «Bahasa Dictionary». Пополнять только в Notion; `base.js` не править вручную; `id` не трогать.
